# Mac 文件搜索工具参考资料：Everything + AnyTXT

资料来源：

- Everything FAQ：<https://www.voidtools.com/zh-cn/faq/>
- AnyTXT Searcher：<https://anytxt.net/google-desktop-search-alternative/>

目标：整理两个 Windows 文件搜索工具的功能与界面风格，为一个融合二者能力的 Mac 版本工具提供产品和 UI 参考。

## 1. 产品定位对比

### Everything

Everything 的核心定位是“极快的本地文件名搜索工具”。它主要通过建立文件名索引，让用户输入关键词后几乎实时获得文件和文件夹结果。

它更像是 Finder 搜索、Spotlight 文件名搜索、命令启动器之间的一个高性能替代品。用户打开它时，预期是快速找到一个文件或文件夹，而不是阅读文件内容。

### AnyTXT Searcher

AnyTXT 的核心定位是“本地文件全文搜索工具”。它不仅找文件名，还会索引文件内容，支持在 Office、PDF、电子书、纯文本、代码等文档内部搜索关键词。

它更像是 Google Desktop Search 的替代品：用户不一定记得文件名，只记得文件里出现过的一句话、术语、合同编号、代码片段或正文内容。

### 融合版 Mac 工具

如果融合两者，Mac 版本的核心定位可以是：

> 一个本地优先、速度像 Everything、内容理解像 AnyTXT、体验像原生 macOS 的文件搜索与预览工具。

它不应该只是“文件名搜索 + 全文搜索”的功能堆叠，而应该围绕 Mac 用户的真实流程设计：

1. 快速找到文件。
2. 快速判断是不是目标文件。
3. 快速预览内容命中位置。
4. 快速打开、复制路径、在 Finder 中显示，或执行后续操作。

## 2. 功能列表

### Everything 功能

Everything 的功能重点可以归纳为：

| 模块 | 功能 |
| --- | --- |
| 文件名索引 | 快速索引本地卷上的文件和文件夹名称 |
| 即时搜索 | 输入关键词后实时过滤结果 |
| 低资源占用 | 安装包小、运行轻、索引速度快 |
| 文件夹/文件搜索 | 同时搜索文件和文件夹 |
| 排序与列表 | 按名称、路径、大小、修改时间等字段查看 |
| 高级查询 | 支持通配符、布尔条件、正则等高级搜索方式 |
| 路径限定 | 可按路径、文件类型、文件夹范围缩小搜索 |
| 索引维护 | 支持自动更新索引，保持与文件系统变化同步 |
| 文件操作 | 打开、定位、复制路径等基础操作 |
| 远程/服务能力 | Windows 生态中支持 HTTP/ETP 等扩展访问方式 |

对 Mac 融合版的启发：

- 首页搜索必须极快，先满足“文件名命中”的即时反馈。
- 文件名搜索结果不能被复杂内容检索拖慢。
- 高级搜索可以存在，但不应该压迫主界面。
- 结果列表要密度高、可排序、可键盘操作。

### AnyTXT 功能

AnyTXT 的功能重点可以归纳为：

| 模块 | 功能 |
| --- | --- |
| 全文索引 | 索引文件内容，而不只是文件名 |
| 多格式支持 | 支持 PDF、Word、Excel、PowerPoint、文本、电子书、代码等多种文件 |
| 内容关键词搜索 | 输入关键词后搜索正文命中 |
| 预览命中内容 | 在结果中查看关键词所在片段 |
| 文件预览 | 帮助用户判断搜索结果是否正确 |
| 文件类型过滤 | 按文档、PDF、表格、代码、图片等类型筛选 |
| 索引管理 | 配置索引目录、排除目录、更新索引 |
| 批量结果 | 在大量文档中定位包含关键词的文件 |
| 本地隐私 | 搜索和索引发生在本地机器 |

对 Mac 融合版的启发：

- 需要把“文件名搜索”和“内容搜索”区分清楚，但体验上不能割裂。
- 内容搜索应该有命中片段，而不是只给文件名。
- 预览区应该是核心区域，尤其适合 PDF、文档、Markdown、代码文件。
- 索引状态必须可见，比如正在扫描、待解析数量、失败文件数量。

## 4. UI 设计风格分析

### Everything 的 UI 风格

Everything 是典型 Windows 工具型界面：

- 高密度表格列表。
- 顶部一个搜索框。
- 菜单栏和工具栏偏传统。
- 几乎没有装饰。
- 信息优先，视觉退后。
- 结果区域占据大部分空间。
- 操作路径直接，学习成本低。

优点：

- 快。
- 清楚。
- 低干扰。
- 适合大量结果浏览。
- 专业用户容易上手。

缺点：

- 视觉比较旧。
- 不像现代 Mac 应用。
- 缺少预览和上下文引导。
- 新用户可能觉得冷冰冰。

Mac 版本应该继承：

- 高密度结果列表。
- 顶部搜索为核心。
- 键盘优先操作。
- 结果字段清晰。

Mac 版本应该避免：

- 传统 Windows 菜单感。
- 过重表格边框。
- 拥挤的小按钮。
- 缺少视觉层级。

### AnyTXT 的 UI 风格

AnyTXT 更接近“内容搜索工具”：

- 搜索框仍然是核心。
- 结果不仅显示文件，还强调内容命中。
- 需要承载文件类型、路径、命中片段、预览等更多信息。
- 信息复杂度比 Everything 高。

优点：

- 更适合“我记得内容但不记得文件名”的场景。
- 命中片段能显著减少打开错误文件的次数。
- 文件类型过滤对文档搜索很有价值。

缺点：

- 界面容易变复杂。
- 内容搜索天然比文件名搜索慢，容易让用户觉得不够轻。
- 如果预览和结果列表没有分层，会显得拥挤。

Mac 版本应该继承：

- 内容命中片段。
- 文件类型过滤。
- 索引配置。
- 预览辅助判断。

Mac 版本应该避免：

- 把所有筛选项同时堆在主界面。
- 让索引状态打断搜索体验。
- 内容结果和文件名结果混在一起却没有标识。

## 5. Mac 融合版 UI 方向建议

### 核心设计原则

1. 搜索框是产品中心。
2. 结果列表要保持 Everything 的速度感和密度。
3. 预览区要承接 AnyTXT 的内容判断能力。
4. 高级筛选折叠到侧边栏、popover 或 token 查询中。
5. 遵循 macOS 原生结构：toolbar、sidebar、detail、inspector、Quick Look、Settings。

### 推荐主窗口结构

```text
+--------------------------------------------------------------+
| Toolbar: 搜索框                         模式  筛选  索引状态 |
+-------------+------------------------------+-----------------+
| Sidebar     | Results                      | Preview         |
|             |                              |                 |
| 全部         | 文件名/内容结果列表            | 文件预览          |
| 文档         | 名称                          | 命中片段          |
| PDF         | 路径                          | 元数据            |
| 图片         | 类型 / 修改时间 / 大小          | 快捷操作          |
| 代码         |                              |                 |
| 收藏目录     |                              |                 |
+-------------+------------------------------+-----------------+
```

### 推荐交互模型

| 区域 | 设计建议 |
| --- | --- |
| 搜索框 | 放在 toolbar 中，支持自然输入和 token 语法 |
| 模式切换 | 使用 segmented control：文件名 / 内容 / 混合 |
| 左侧栏 | 放范围和类型，不放复杂参数 |
| 中间结果 | 高密度列表，支持键盘上下选择 |
| 右侧预览 | 显示 Quick Look、文本命中、PDF 页码、元数据 |
| 底部状态 | 显示索引状态、结果数量、搜索耗时 |
| 设置页 | 独立 Settings 窗口，管理索引目录、排除规则、解析器 |

### 视觉风格

适合的方向：

- 原生 macOS。
- 专业工具感。
- 安静、克制、信息密度高。
- 类似 Finder + Mail + Raycast + Preview 的结合。
- 不做营销型大卡片。
- 不做强烈渐变。
- 不做过度圆角。

建议视觉关键词：

- 清晰
- 快
- 本地
- 可信
- 专业
- 低打扰
- 可扫描

建议颜色：

- 使用系统自适应背景和材料，支持浅色/深色模式。
- 主色可以选择克制的蓝色或青绿色，用于选中、命中高亮、状态。
- 命中关键词用淡黄色或系统强调色，不要大面积铺色。
- 错误/权限问题使用系统红色，但只在状态和详情里出现。

建议字体：

- 主界面使用系统字体 SF Pro。
- 文件路径、代码命中、查询语法使用等宽字体 SF Mono。
- 不建议使用装饰性字体。

### 需要重点打磨的 UI 细节

1. 搜索结果行
   - 名称要最突出。
   - 路径要可读但弱化。
   - 命中类型要明确：文件名命中 / 内容命中。
   - 修改时间、大小、类型不要抢注意力。

2. 内容命中片段
   - 关键词高亮。
   - 显示上下文，不要只显示孤立词。
   - 多个命中可以折叠。

3. 预览区
   - 选中文件后立即显示预览。
   - 对不支持预览的文件显示元数据和可执行动作。
   - PDF/文本类文件要能跳到命中位置。

4. 索引状态
   - 不要弹窗打扰。
   - 在 toolbar 或状态区显示“正在索引 / 待解析 N / 失败 N”。
   - 点击后进入索引详情。

5. 空状态
   - 第一次启动：引导用户选择索引目录。
   - 无结果：给出缩小/扩大搜索范围建议。
   - 无权限：明确告诉用户去系统设置授权。

## 6. 初始版本建议

第一版不要试图完整复制 Everything 和 AnyTXT 的全部能力。建议先做这个闭环：

```text
选择索引目录
→ 建立文件名索引
→ 建立基础文本内容索引
→ 输入关键词
→ 显示文件名和内容结果
→ 选中结果
→ 右侧预览
→ 打开 / Finder 中显示 / 复制路径
```

第一版 UI 可以只包含：

- toolbar 搜索框
- 搜索模式切换
- 左侧类型/范围 sidebar
- 中间结果列表
- 右侧预览
- 设置页中的索引目录和排除规则
- 索引状态提示

这会比一开始加入 OCR、语义搜索、复杂高级语法更稳。

## 7. 一句话产品方向

> 做一个 Mac 上的本地文件搜索工作台：Everything 负责速度，AnyTXT 负责内容，macOS 负责体验。
